Architecture

The design decisions that separate OpenRCAD from legacy kernels — and the few that look unusual until you see the bug they prevent.

Hard rules

  • #![forbid(unsafe_code)] everywhere. Every crate. No exceptions. All data structures and algorithms are safe Rust.
  • Geometry is owned enums, not Box<dyn Trait>. GeomCurve and GeomSurface are sum types stored by value inside topology — so topology is Clone, Serialize, and lifetime-free.
  • Minimal dependencies. Only serde for core crates; slotmap in topology and rayon in meshing. No C/C++ anywhere.

1 · Arena-based B-Rep

B-Rep graphs are inherently cyclic (Face → Loop → Edge → Vertex → …). Safe Rust cannot represent pointer cycles without Rc/Arc + locks, which destroy cache locality. OpenRCAD stores every entity in flat generational arenas (slotmap) and references them by Copy-able index handles (VertexId, EdgeId, …). Adjacency is arrays of indices, giving O(1) access, contiguous layout, and lock-free concurrent reads via rayon.

2 · BVH spatial acceleration

Every geometric query (intersection, closest point, containment) is accelerated by a bounding-volume hierarchy of axis-aligned boxes, built with the Surface-Area Heuristic and rebuilt lazily on topology change. This takes intersection and boolean complexity from O(n²) toward O(n log n).

3 · Robust numerics

Floating-point error is the #1 cause of boolean and fillet failures in legacy kernels. OpenRCAD uses Shewchuk-style adaptive exact predicates (fast floating-point, promoted to exact only when the sign is ambiguous) for orientation tests, and adaptive subdivision with interval bounding so intersection curves are proven topologically correct before they enter the B-Rep.

4 · Tolerant modeling

Imported CAD data has small gaps. A single global tolerance is too rigid, so each Vertex and Edge stores its own local tolerance — the radius of its uncertainty sphere/tube — and every adjacency and sewing check respects it. This is what lets the kernel process "dirty" STEP/IGES geometry.

5 · NURBS done right

Curves and surfaces evaluate with de Boor's algorithm (local support, O(k²)), refine with Boehm knot insertion, and take derivatives directly from de Boor — never naive basis summation or finite differences. The new ToroidalSurface joins planes, cylinders, cones, spheres, Bézier/Gregory, offset, and ruled surfaces as first-class analytic variants.

6 · Local Euler operators

Local edits (fillets, chamfers, face deletions) use atomic Euler operators (MEV, MEF, KEV, KEF, …) that surgically modify the B-Rep graph and preserve the Euler–Poincaré invariant V − E + F = 2(S − G) + H — rather than a global boolean sweep.

7 · Blending & offset

The blend system targets the cases that crash other kernels: rolling-ball edge fillets (constant + variable radius), N-sided Gregory-patch corner blends for n-valent vertices, distance/angle chamfers, and offset-surface shelling with self-intersection resolution. The whole-solid builders handle a single box or cylinder primitive at any orientation (the frame is recovered from geometry), each finishing watertight, and the per-edge rolling-ball fillet handles arbitrary planar/analytic edges including boolean results; booleans are watertight and health-validated on partial-imprint and coplanar cases. N-valent corner blends, fillet face overflow, and concave-offset self-intersection resolution remain the active frontier (see Status).

8 · Sewing & healing

Robust import/export needs repair: identify free edges on open shells, match and merge them within per-entity tolerance, and validate the result for manifoldness, orientation consistency, and watertightness. Every blend builder finishes by sewing its faces into one watertight shell.

The layered dependency graph

The workspace is strictly layered and points one way — no cycles between crates.

            openrcad  (facade, re-exports the kernel)
                │
   ┌────────────┼───────────────┬───────────────┐
 primitives   exchange        mesh            algo
   │            │               │               │
   └────────► topo ◄────────────┴───────────────┘
                │   ◄── geom, geom2d
                │
          foundation   (no internal deps — the gp layer)

 document ──► algo + sketch     (parametric history, .zcad)
 render   ──► mesh + topo       (wgpu viewer — outside the facade)

The viewer (render) and truck-compat sit outside the facade so GPU and interop dependencies never reach a core kernel crate. For the canonical, source-of-truth description of these principles, see CLAUDE.md in the repository root.
