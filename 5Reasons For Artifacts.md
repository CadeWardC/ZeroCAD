# Fixing Fillet and Boolean Radiating Line Artifacts

This implementation plan investigates and proposes fixes for the rendering anomalies where lines radiate infinitely across the screen during Boolean operations (Join/Cut) and fillet operations.

---

## 5 Reasons for the Artifacts

### 1. Near-Parallel Edge Intersections in the Polygon Offset Solver (Primary Culprit for 3D)
* **Location**: [mock_kernel.rs:L473-480](file:///c:/Users/cadel/Documents/Coding_Projects/ZeroCAD/zerocad-core/src/mock_kernel.rs#L473-L480) inside `offset_polygon_outward`.
* **Details**: A 3D fillet cutter is built by sweeping a 2D cross-section. The fillet arc meets the flat faces tangentially at `t1_2d` and `t2_2d`. When `offset_polygon_outward` is called to inflate the cutter profile (so it cuts clean through without tangency issues), it computes the intersection of consecutive edges. Because the arc's end segment is virtually collinear with the face segment, the denominator `denom` in `intersect` is extremely close to zero. If it is slightly larger than the hardcoded tolerance `1.0e-9` (e.g., `1.1e-9`), the intersection offset `t` becomes massive. This projects the vertex thousands of millimeters away, creating an extreme "spike" vertex in the 3D cutter. Subtracting this from the main body leaves corrupted mesh vertices that project to infinity, resulting in fanning lines across the screen.
* **Fix**: Implement a safety check or a miter limit. If the distance from the original vertex to the offset intersection exceeds a sensible threshold (e.g., `grow * 2.5`), or if the angle between the two edges is nearly 180°, fall back to the unmitered offset vertex (`off_pt[i]`) or bevel the corner.

### 2. Snap-to-Nearest-Vertex layering on 2D Sketch Fillets (Primary Culprit for 2D)
* **Location**: [sketch.rs:L339-353](file:///c:/Users/cadel/Documents/Coding_Projects/ZeroCAD/zerocad-core/src/sketch.rs#L339-L353) inside `apply_corner_mod`.
* **Details**: In 2D sketch mode, when a user applies a fillet to a corner, the solver snaps to the nearest vertex. If the user interactively drags the fillet size or applies multiple fillets in sequence, the snap-to-nearest-vertex logic might snap to one of the newly generated arc endpoints of an existing fillet instead of the original corner vertex (which has been trimmed away). This results in applying a fillet *on top* of an existing fillet, creating concentric overlapping arcs (the fanning arcs in Picture 1) and leaving trimmed segment remnants shooting off in weird directions.
* **Fix**: Ensure that topological IDs are maintained for corners, or prevent applying new corner modifiers to vertices that are already part of an existing fillet arc. Replace or update existing modifiers at the same location instead of appending them.

### 3. Missing 3D Line Clipping Against the Camera's Near Plane
* **Location**: [render.rs:L854-893](file:///c:/Users/cadel/Documents/Coding_Projects/ZeroCAD/zerocad-gui/src/render.rs#L854-L893) inside `render_viewport`.
* **Details**: The 3D line projection code `project_3d` transforms 3D coordinate vertices to 2D screen positions. If any edge of the model has vertices that lie behind the camera or are extremely far away (such as those generated by the boolean solver failures above), dividing by `dist - final_z` can yield division-by-near-zero or extreme coordinate values. Without clipping the 3D line segment to the camera's view frustum/near plane *prior* to projection, these projected screen points will shoot out infinitely to the margins of the screen.
* **Fix**: Clip all 3D line segments against the near clipping plane in 3D camera space before projecting their endpoints to 2D screen coordinates.

### 4. Watertightness and Coplanar Face Failures in the CAD Kernel (`truck`)
* **Location**: [mock_kernel.rs:L560-587](file:///c:/Users/cadel/Documents/Coding_Projects/ZeroCAD/zerocad-core/src/mock_kernel.rs#L560-L587) inside `guarded_boolean`.
* **Details**: CAD kernels are highly sensitive to tangent or coplanar face intersections. When a cylinder is cut or joined exactly coplanar to a box face, precision tolerances can cause the B-Rep boundary curves to fail to close cleanly. This can leave open shell gaps or degenerate faces with zero area, which project edges at undefined/garbage positions.
* **Fix**: Ensure that cut tools are slightly oversized (overshooting the target face by a tiny fraction like `0.01mm`) to enforce a clean transversal intersection instead of a coplanar face-to-face match.

### 5. Inadequate Crease Edge Filtering for Tessellated Cylinders
* **Location**: [mock_kernel.rs:L1276-1291](file:///c:/Users/cadel/Documents/Coding_Projects/ZeroCAD/zerocad-core/src/mock_kernel.rs#L1276-L1291) inside `mesh_feature_edges`.
* **Details**: During Boolean operations, the new curved walls are approximated by facets. The wireframe generator tries to hide the lengthwise seams by comparing the adjacent face normals against a crease angle threshold (`CREASE_COS: f32 = 0.95`). If the triangulation has uneven face normals, or if the threshold is too restrictive, the renderer fails to filter out these internal seams, causing the faceted cylinder walls to display as a series of long lines fanning down the cylindrical surface.
* **Fix**: Recalculate face normals from the actual geometric surfaces before crease filtering, or slightly loosen the threshold to suppress seam lines on faceted cylinders.

## Most Likely Culprit

### For the 3D Join/Cut Spikes (Pictures 2 & 3):
The most likely culprit is **Cause #1 (Near-Parallel Edge Intersections)**. When the cylinder is booleaned into the box, the kernel evaluates the fillet cutters on the box's edges. Because the tangent lines of the fillet cutter cross-section are nearly parallel to the box faces, the offset calculation in `offset_polygon_outward` divides by a value close to zero, generating a vertex at an extreme coordinate. This massive vertex corrupts the output body's B-Rep and wireframe mesh, creating lines that stretch infinitely across the viewport.

### For the 2D Fillet Artifacts (Picture 1):
The most likely culprit is **Cause #2 (Snap-to-Nearest-Vertex layering)**. When dragging or updating the corner radius, the tool snaps to the newly created vertices of the fillet arc rather than keeping the original logical corner, resulting in multiple nested arcs and stray tangent lines.
